# S023.1 - Database + Backend Foundation

## Epic

Epic 5 - Lessons and Learner Experience (parent: [S023](./00-overview.md))

## Status

Planned

## Goal

Create the `lessons` and `lesson_items` tables, SQLAlchemy models, and migrations so later slices can build APIs and UI on a stable schema.

## Dependencies

- S021 concept completion (concepts and concept_texts exist and are stable).
- Existing migration pattern in `backend/` (Flask-Migrate or project equivalent).

## Scope

### Tables

Implement fields from [00-overview.md](./00-overview.md#data-model-mvp).

**`lessons`**

- UUID PK, timestamps.
- Unique `slug`.
- `status`: `draft`, `published`, `archived`.
- `difficulty`: same check constraint as concepts — `beginner`, `intermediate`, `advanced`.
- Nullable cover image fields (mirror concept image pattern if Cloudinary fields already exist).

**`lesson_items`**

- UUID PK, timestamps.
- FK `lesson_id` → `lessons.id` **ON DELETE CASCADE**.
- FK `concept_id` → `concepts.id` **ON DELETE RESTRICT** (do not delete concepts referenced by items).
- `type`: `VOCABULARY`, `PHRASE`, `AUDIO_LISTEN`, `CULTURAL_NOTE`.
- `content_json`: JSON/JSONB column, default `{}`.
- Unique constraint on `(lesson_id, order_index)`.

### Models

- Add models under `backend/app/models/` following Languages/Concepts patterns.
- Relationships: `Lesson.items`, `LessonItem.lesson`, `LessonItem.concept` (optional).

### Validation helpers (model or schema layer)

- `concept_id` required when `type` in (`VOCABULARY`, `PHRASE`, `AUDIO_LISTEN`).
- `concept_id` optional when `type = CULTURAL_NOTE`.
- `content_json.noteTextEn` and `noteTextFr` required when `type = CULTURAL_NOTE` (validate at API layer).

## Out Of Scope

- HTTP routes.
- Frontend types.
- Seed data.
- Import from CSV.

## Acceptance Criteria

- Migration applies cleanly on fresh and existing dev databases.
- Models import without circular dependency errors.
- Unique slug and per-lesson order_index constraints enforced.
- Backend pytest includes at least model/constraint smoke tests if the repo pattern supports it.
- Parent overview data model matches implemented columns.
